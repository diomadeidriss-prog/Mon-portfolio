Pourquoi le C est essentiel en sécurité

La majorité des vulnérabilités critiques — buffer overflow, use-after-free, format string — se produisent dans du code C ou C++. Pour les comprendre, les détecter dans des audits de code, et les exploiter ou les corriger, il faut maîtriser le langage à fond.

Ce projet consistait à recoder les fonctions fondamentales de la libc (la bibliothèque standard C) depuis zéro, sans utiliser les fonctions originales. Un exercice brutal mais incroyablement formateur.

Comprendre comment strcpy() fonctionne en interne, c'est comprendre pourquoi elle est vulnérable aux buffer overflows et comment les attaquants l'exploitent.

Le modèle mémoire en C

La première chose à maîtriser est comment C organise la mémoire d'un programme. Cette compréhension est directement liée aux attaques de type stack smashing et heap exploitation.

Fonctions réimplémentées

Voici quelques-unes des fonctions recodées, avec leur lien direct avec la sécurité :

/*
** ft_strlen — Compte les caractères jusqu'au \0
** Vecteur: si la chaîne n'est pas null-terminée → lecture out-of-bounds
*/
size_t ft_strlen(const char *s)
{
    size_t i = 0;
    while (s[i] != '\0')
        i++;
    return i;
}

/*
** ft_strcpy — Copie une chaîne dans un buffer destination
** ⚠ DANGEREUX: pas de vérification de taille → buffer overflow classique
** C'est exactement pourquoi on utilise strncpy() ou strlcpy() en prod
*/
char    *ft_strcpy(char *dest, const char *src)
{
    size_t i = 0;
    while (src[i] != '\0')
    {
        dest[i] = src[i];  /* Écrase sans vérifier → overflow possible */
        i++;
    }
    dest[i] = '\0';
    return dest;
}
/*
** ft_memset — Remplit un bloc mémoire avec une valeur
** Utilisée pour effacer des données sensibles (clés, mots de passe)
*/
void    *ft_memset(void *b, int c, size_t len)
{
    unsigned char *ptr = (unsigned char *)b;
    size_t         i = 0;

    while (i < len)
        ptr[i++] = (unsigned char)c;
    return b;
}

/*
** ft_calloc — Alloue et initialise à zéro (évite les données résiduelles)
** Important: toujours vérifier le retour de malloc != NULL
*/
void    *ft_calloc(size_t count, size_t size)
{
    void *ptr;

    if (count == 0 || size == 0) return malloc(0);
    if (count > SIZE_MAX / size) return NULL; /* Integer overflow check */
    ptr = malloc(count * size);
    if (!ptr) return NULL;
    ft_memset(ptr, 0, count * size);
    return ptr;
}

Le lien avec les vulnérabilités

Coder ces fonctions m'a permis de comprendre exactement pourquoi certaines pratiques génèrent des failles :

  • Buffer Overflow — strcpy() sans vérification de taille écrase l'adresse de retour sur la stack → contrôle du flux d'exécution par un attaquant.
  • Use-After-Free — Utiliser un pointeur après free() → comportement indéfini, exploitable pour corrompre le heap.
  • Integer Overflow — Un calcul de taille comme count * size peut dépasser SIZE_MAX et allouer moins que prévu → heap overflow.
  • Format String — printf(user_input) sans format string → lecture/écriture arbitraire en mémoire.

Exemple concret : un gets(buffer) en C est toujours vulnérable à un buffer overflow — il n'y a aucun moyen de l'utiliser de façon sécurisée. Cette fonction est retirée de C11. La comprendre c'est comprendre CVE-2021-3156 (sudo), Log4Shell, et des centaines d'autres.

Ce que ce projet m'a apporté

La maîtrise du C bas-niveau est une compétence rare et très valorisée en cybersécurité. Elle me permet aujourd'hui de lire et analyser du code C dans des audits, de comprendre les CVEs au niveau technique, et d'écrire des exploits ou des patchs de sécurité.

Cette formation a directement nourri mes projets de pentest : comprendre les primitives mémoire me permet d'analyser des binaires et d'identifier des vecteurs d'attaque invisibles pour ceux qui n'ont pas cette base.
